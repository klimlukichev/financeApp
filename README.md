# Финансы / FinanceApp

![icon](https://github.com/klimlukichev/financeApp/blob/master/icon.png)

**FinanceApp** — локальное Android-приложение для учёта личных расходов. Приложение хранит данные на устройстве пользователя, не требует регистрации, не использует банковские API и не передаёт финансовую информацию на внешний сервер.

Основной сценарий работы: пользователь добавляет операции вручную или импортирует расходы из PDF-выписок Т-Банка и Сбербанка, после чего приложение автоматически распределяет операции по категориям, строит аналитику за месяц, контролирует недельный бюджет и позволяет экспортировать данные.

## Возможности

- Ручное добавление расходов с суммой, датой, категорией и примечанием.
- Редактирование и удаление ранее созданных операций.
- Импорт расходных операций из PDF-выписок Т-Банка и Сбербанка.
- Автоматическое определение банка по содержимому PDF-выписки.
- Автоматическая категоризация операций по ключевым словам, категории банка и пользовательским правилам.
- Запоминание пользовательских исправлений категорий.
- Защита от повторного импорта одинаковых операций.
- Список операций за выбранный календарный месяц.
- Навигация между месяцами.
- Суммарный расход за месяц.
- Диаграмма расходов по категориям.
- Исключение внутренних переводов из аналитики.
- Недельный бюджет и уведомление при превышении лимита.
- Ежедневные напоминания о внесении расходов.
- Экспорт операций за месяц в CSV.
- Формирование PDF-отчёта за месяц.
- Светлая и тёмная тема.
- Переключение языка интерфейса: русский / английский.
- Локальное хранение данных без обязательного подключения к интернету.

## Принципы работы

FinanceApp построен вокруг локального учёта расходов:

1. Пользователь вручную добавляет операцию или выбирает PDF-выписку через системный file picker Android.
2. При импорте приложение извлекает текст из PDF-файла и выбирает подходящий парсер банка.
3. Из выписки формируется список банковских операций.
4. В базу сохраняются только расходы.
5. Каждая операция получает категорию.
6. Для импортированных операций вычисляется `importHash`, чтобы повторная загрузка той же выписки не создавала дубликаты.
7. Главный экран получает данные из Room через `Flow` и автоматически обновляет список операций, итоговую сумму и диаграмму.
8. Аналитика дополнительно фильтрует внутренние переводы между собственными счетами.

Приложение не подключается к банковским API. Пользователь сам выгружает PDF-выписку из банка и выбирает её на устройстве.

## Архитектура

Проект использует многослойную архитектуру: Clean Architecture + MVVM + Repository.

```text
ui
  -> domain
      -> data
```

Основные правила:

- UI не работает напрямую с Room, DataStore и PDF-парсерами.
- ViewModel вызывает use case и репозитории через интерфейсы.
- Доменный слой содержит бизнес-правила и не зависит от Android UI.
- Data-слой инкапсулирует Room, DataStore, PDF-парсинг и мапперы.
- Зависимости собираются в `di/AppModule.kt` через Koin.

### Слои

| Слой | Пакеты | Ответственность |
|---|---|---|
| UI | `ui`, `ui.home`, `ui.theme`, `ui.locale` | Compose-экран, диалоги, состояние интерфейса, тема, локализация |
| Domain | `domain.model`, `domain.usecase`, `domain.analytics`, `domain.repository` | Модели, бизнес-логика, интерфейсы репозиториев, аналитика |
| Data | `data.local`, `data.repository`, `data.importing`, `data.mapper` | Room, DataStore, PDF-парсинг, реализации репозиториев |
| DI | `di` | Регистрация зависимостей Koin |
| Notifications | `notifications` | Уведомления, каналы, WorkManager |

## Модель данных

### `categories`

Справочник категорий расходов.

| Поле | Тип | Назначение |
|---|---|---|
| `id` | `Long` | Первичный ключ |
| `name` | `String` | Название категории |
| `iconResId` | `Int` | Ресурс иконки |
| `colorInt` | `Int` | Цвет категории для диаграмм |
| `isDefault` | `Boolean` | Признак предустановленной категории |

Категории создаются при первом запуске через `DatabaseSeeder`.

### `transactions`

Операции пользователя.

| Поле | Тип | Назначение |
|---|---|---|
| `id` | `Long` | Первичный ключ |
| `amount` | `Double` | Сумма операции |
| `date` | `Long` | Дата в формате Unix timestamp |
| `categoryId` | `Long` | Внешний ключ на `categories` |
| `note` | `String?` | Примечание или описание операции |
| `type` | `String` | Тип операции, по умолчанию `EXPENSE` |
| `sourceBank` | `String?` | Банк-источник импортированной операции |
| `sourceDescription` | `String?` | Исходное описание из выписки |
| `importHash` | `String?` | Хеш для защиты от повторного импорта |

Для `transactions` заданы индексы:

- `categoryId`;
- `date`;
- уникальный индекс по `importHash`.

Связь с категорией защищена внешним ключом с `ForeignKey.RESTRICT`: категорию нельзя удалить, если с ней связаны операции.

## Импорт PDF-выписок

Импорт выполняется полностью локально.

Поддерживаемые форматы:

- PDF-выписка Т-Банка.
- PDF-выписка Сбербанка по счёту дебетовой карты.

Компоненты импорта:

| Компонент | Назначение |
|---|---|
| `PdfTextExtractor` | Извлекает текст из PDF через PdfBox-Android |
| `BankStatementParser` | Общий интерфейс парсеров |
| `BankStatementParserFactory` | Выбирает парсер по содержимому текста |
| `TBankStatementParser` | Разбирает выписки Т-Банка |
| `SberStatementParser` | Разбирает выписки Сбербанка |
| `OfflineBankStatementImportRepository` | Объединяет извлечение текста и парсинг |
| `ImportBankStatementUseCase` | Отбирает расходы, категоризирует и сохраняет операции |

Общий поток импорта:

1. Пользователь выбирает PDF-файл через системный picker.
2. Приложение получает `InputStream`.
3. `PdfTextExtractor` извлекает текст.
4. `BankStatementParserFactory` выбирает парсер.
5. Парсер возвращает список `BankStatementTransaction`.
6. `ImportBankStatementUseCase` оставляет только расходы.
7. Каждая операция категоризируется.
8. Формируется `importHash`.
9. Операции сохраняются через `insertAllIgnore`.
10. Пользователь получает результат: распознано, импортировано, пропущено.

### Защита от дубликатов

Для импортированных операций вычисляется `importHash`.

В хеш входят:

- банк;
- дата;
- сумма;
- тип операции;
- описание;
- порядковый номер операции в выписке.

Хеш вычисляется через SHA-256. В Room для `importHash` создан уникальный индекс. Повторный импорт той же выписки не создаёт повторные операции: `insertAllIgnore` пропускает записи с уже существующим хешем.

## Категоризация и аналитика

### Категоризация

Категоризация выполняется классом `TransactionCategorizer` и use case `CategorizeByKeywordsUseCase`.

Источники для выбора категории:

- пользовательские правила;
- категория банка;
- ключевые слова в описании операции;
- эвристики для типовых расходов.

Пользовательские правила имеют высокий приоритет. Они создаются через `RememberCategoryCorrectionUseCase`, когда пользователь вручную исправляет категорию операции.

Для категории "Переводы" действует дополнительное правило: одного упоминания СБП недостаточно. Категория назначается только при наличии слова "перевод" или его словоформ.

### Аналитика

Аналитика строится через `GetCategorySpendingUseCase`:

1. Определяется период выбранного месяца.
2. Загружаются операции и категории.
3. Операции связываются с названиями категорий.
4. Применяется `AnalyticsExpenseFilter`.
5. Данные агрегируются через `CategorySpendingAggregator`.
6. UI получает список `CategorySpending` и отображает диаграмму.

### Фильтрация внутренних переводов

`AnalyticsExpenseFilter` исключает операции, которые не являются реальными тратами:

- явные внутренние переводы по тексту;
- операции с признаками перевода между своими счетами;
- промежуточные звенья цепочек переводов между банками на одинаковую сумму в один день.

Фильтр применяется:

- в диаграмме расходов;
- в итоговой сумме за месяц;
- при проверке недельного бюджета;
- при формировании PDF-отчёта.

## Бюджет и уведомления

Пользователь может задать недельный лимит бюджета. Значение хранится в DataStore.

Проверка выполняется в `CheckWeeklyBudgetUseCase`:

1. Загружаются настройки бюджета.
2. Определяется текущая неделя.
3. Загружаются операции за неделю.
4. Исключаются внутренние переводы.
5. Сумма расходов сравнивается с лимитом.
6. Если лимит превышен и уведомление за неделю ещё не отправлялось, формируется `BudgetExceededInfo`.

Уведомления реализованы через:

- `FinanceNotificationManager`;
- `DailyReminderScheduler`;
- `DailyReminderWorker`;
- WorkManager.

Для Android 13+ используется разрешение:

```xml
<uses-permission android:name="android.permission.POST_NOTIFICATIONS" />
```

Если пользователь не выдаст разрешение, основной функционал продолжит работать, но системные уведомления могут не отображаться.

## Экспорт данных

### CSV

`ExportTransactionsUseCase` формирует CSV-файл за выбранный месяц.

Файл содержит:

- `id`;
- `date`;
- `amount`;
- `category_id`;
- `category`;
- `note`.

CSV записывается в UTF-8 с BOM, чтобы файл корректно открывался в табличных редакторах.

### PDF

`ExportPdfReportUseCase` формирует PDF-отчёт средствами Android `PdfDocument`.

Отчёт включает:

- заголовок и месяц;
- итоговые суммы;
- диаграмму расходов;
- таблицу операций за месяц;
- дополнительные страницы при большом числе операций.

Экспорт использует системный диалог создания файла. Приложение получает доступ только к выбранному пользователем месту сохранения.

## Ограничения

- Импорт поддерживает только PDF-выписки Т-Банка и Сбербанка в ожидаемых форматах.
- Из PDF-выписок в текущей логике сохраняются только расходы.
- Банковские API не используются.
- Синхронизация между устройствами не реализована.
- Резервное копирование пользовательских данных не реализовано.
- UI-автотесты и интеграционные тесты не добавлены.
- Валюта учёта ориентирована на рубли.
